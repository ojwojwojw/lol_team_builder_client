# 장애 리포트

날짜: 2026-05-05  
프로젝트: LOL Team Builder  
주제: 배포 서버에서 최근 매치 조회가 비어 보이던 문제

## 1. 증상

- Cloud Run에 배포된 서버에서 계정 검색은 정상 동작했다.
- Firestore에도 경기 데이터가 저장되어 있었다.
- 그런데 메인 클라이언트에서 특정 유저를 선택했을 때 `최근 매치` 목록이 비어 보였다.
- 로컬 개발 환경에서는 같은 흐름이 재현되지 않거나, 원인 파악이 더 어려웠다.

## 2. 직접 원인

직접 원인은 `matches` 컬렉션과 `match_participants` 컬렉션의 역할 차이, 그리고 기존 적재 스킵 로직의 조합이었다.

- `matches`
  - 경기 원본 상세 문서를 저장하는 컬렉션
- `match_participants`
  - 최근 매치 목록 조회를 빠르게 하기 위한 참가자 단위 인덱스 컬렉션

최근 매치 조회 API는 실제로 `match_participants`를 기준으로 데이터를 읽는다.  
그런데 과거에 이미 `matches` 문서가 저장된 경기들은, 이후 적재 시 `이미 존재하는 경기`로 판단되어 상세 재조회 없이 건너뛰었다.

이 상태에서 다음과 같은 불일치가 생길 수 있었다.

- `matches` 문서는 존재함
- 하지만 `match_participants` 문서는 없거나, 일부만 있거나, 현재 조회 스키마와 맞지 않음

결과적으로 Firestore 안에는 데이터가 있는데도, 클라이언트가 보는 최근 매치 목록은 비어 보이는 현상이 발생했다.

## 3. 왜 배포 환경에서 더 잘 드러났는가

배포 환경에서는 Firestore 실제 데이터가 누적되면서, 과거에 저장된 `matches` 문서와 현재 조회 구조 사이의 차이가 더 크게 드러났다.

특히 다음 두 조건이 겹치면 증상이 쉽게 나타난다.

- 과거에 저장된 `matches` 문서가 이미 있음
- 이후 적재 작업이 `matches` 존재 여부만 보고 스킵함

이 경우 새 적재를 여러 번 수행해도 `match_participants` 인덱스가 복구되지 않아서, 조회는 계속 실패할 수 있다.

## 4. 해결책

두 단계로 해결했다.

### 4-1. 최근 매치 조회 쿼리 완화

배포 환경에서 Firestore 복합 인덱스 의존성을 줄이기 위해, 최근 매치 조회를 다음 방식으로 바꿨다.

- Firestore에서는 `where(...)` 조건으로만 가져온다
- 정렬은 서버 Python에서 `game_start_timestamp` 기준으로 수행한다
- 마지막에 `limit`를 적용한다

이 수정은 배포 환경에서 Firestore 인덱스 민감도를 낮추는 역할을 한다.

### 4-2. 참가자 인덱스 재구성 로직 추가

기존 경기 문서는 있는데 참가자 인덱스가 없는 경우를 복구하기 위해 다음 로직을 추가했다.

- `matches` 문서가 이미 있더라도 완전히 무시하지 않는다
- 해당 경기의 `participants` 배열을 읽는다
- `match_participants` 컬렉션에 참가자 인덱스를 다시 생성한다

즉, 새 경기만 저장하는 방식에서

- 새 경기는 정상 저장
- 기존 경기는 참가자 인덱스를 재생성

하도록 보강했다.

## 6. JWT / 권한 문제 여부 점검

결론부터 말하면, 이번 문제는 `JWT 토큰`이나 `권한 부족`이 직접 원인은 아니었다.

근거는 다음과 같다.

### 6-1. 매치 조회 라우터는 인증 보호가 걸려 있음

`server/backend/controllers/match_controller.py` 에서 매치 조회 라우터는 다음 의존성을 사용한다.

- `Depends(get_current_user)`

즉 토큰이 없거나 잘못되면, 최근 매치 조회 API는 정상 응답 대신 인증 에러를 반환해야 한다.

### 6-2. JWT 검증 실패 시 명확한 401 에러가 발생함

`server/backend/security.py` 의 흐름상 다음 경우에는 바로 실패한다.

- Bearer 토큰 없음
- 토큰 만료
- 토큰 서명 불일치
- 토큰의 `sub`에 해당하는 사용자가 Firestore에 없음
- 사용자가 비활성 상태

이 경우 응답은 비어 있는 매치 목록이 아니라 `401 Unauthorized`가 된다.

### 6-3. 관리자 권한은 최근 매치 조회의 필수 조건이 아님

최근 매치 조회는 `로그인 사용자`면 접근 가능하고, `관리자 전용` 라우트가 아니다.  
따라서 이번 증상이 `관리자가 아니라서 최근 매치가 안 보였다`는 흐름과도 맞지 않는다.

### 6-4. 실제 증상은 인증 성공 후 조회 결과만 비어 있었음

이번 케이스에서는

- 로그인은 유지되고 있었고
- 계정 검색도 정상 동작했고
- Firestore에도 경기 데이터가 있었으며
- 최근 매치 목록만 비어 있었다

이 패턴은 인증 실패보다 `조회 대상 컬렉션/인덱스 불일치`와 훨씬 잘 맞는다.

## 7. 재발 방지 포인트

- 최근 매치 조회가 어떤 컬렉션을 기준으로 동작하는지 문서로 명확히 유지한다.
- `matches`와 `match_participants`의 역할 차이를 운영자가 인지할 수 있게 한다.
- 적재 스킵 조건은 `원본 문서 존재 여부`만 보지 말고, 조회에 필요한 보조 인덱스 상태도 고려한다.
- 필요하면 이후 `repair` 또는 `reindex` 전용 관리자 버튼을 `riot_loader`에 따로 두는 것도 고려할 수 있다.

## 8. 최종 결론

이번 장애는 `JWT`, 인증, 권한 문제가 아니라,  
`Firestore에 저장된 원본 경기 문서와 최근 매치 조회용 참가자 인덱스 간의 불일치` 때문에 발생했다.

해결은 다음 두 축으로 진행되었다.

- 조회 쿼리의 Firestore 인덱스 의존 완화
- 기존 경기 데이터에서 참가자 인덱스를 복구하는 적재 보강

현재는 해당 수정과 재배포 후, 최근 매치 조회가 정상적으로 복구된 상태다.
